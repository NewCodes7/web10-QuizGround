## 해석

### 성공: `useBuffer()` 가 encode 비용을 사실상 제거

msgpackr encode self time이 **23.9s (17.1%) → 2.6s (1.9%)** 로 급감.  
3번 프로파일에서 가장 큰 병목이었던 encode 내부 버퍼 할당 루프가 2MB arena 재사용으로 사라진 결과다.

### 긍정: 서버 전체 여유도 향상

Active 89.3s → 96.4s → **81.2s**, Idle 10.3s → 14.0s → **26.6s**.  
200명 부하에서 Idle이 19%를 확보한 것은 실질적인 여유가 생겼다는 의미다.

### 주의: GC 절대량이 계속 증가

| | GC 절대 시간 |
|--|--|
| 1_기존(JSON) | 20.6s |
| 3_msgpackr | 29.4s |
| 4_buffer재사용 | **30.9s** |

GC(active 기준)가 27.6%까지 올라간 건 **두 가지 요인이 겹친 결과**다.

1. **분모 감소 착시**: Active가 96.4s → 81.2s로 줄면서 비율 분모가 작아짐. GC 절대량(29.4s → 30.9s)은 거의 변하지 않았는데 비율이 올라 보임.
2. **GC 원인이 encode 외에도 남아 있음**: `useBuffer()`가 encode 할당은 제거했지만, `validation` 비용이 1.8s → **3.7s** 로 늘어난 것이 GC 압력의 새 원인으로 부상. `updatePosition` hot path에서 ValidationPipe가 매 포지션 업데이트마다 DTO 객체를 생성하고 버린다.
