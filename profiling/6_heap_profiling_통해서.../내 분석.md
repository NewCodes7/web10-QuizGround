  인과 관계       

  200명 위치 업데이트
    → Position Flush마다 JSON 직렬화 (200 positions × 100ms 주기)                                                                                                               
    → Young gen 빠르게 채움 → Minor GC 6.5/s                                                                                                                                    
    → Heap 단조 증가 (54→68MB)                                                                                                                                                  
      → Old gen 객체 증가 → Minor GC 루트 스캔 비용 증가                                                                                                                        
    → 후반부에 Minor GC 1회 pause가 길어짐 (0.6ms → 9ms+)                                                                                                                       
    → Event Loop Lag P99: 11ms → 238~500ms 폭발                                                                                                                                 
    → BatchProcessor 콜백 지연 → Position Flush P99: 5ms → 474ms                                                                                                                
                                                                                                                                                                                
  CPU 50% 미만인 이유: GC pause는 event loop을 멈추지만 CPU throughput 자체를 크게 소비하지 않습니다. CPU 프로파일러로는 안 잡힙니다.                                           
                  
  후반에 갑자기 나빠지는 이유: Old gen 크기 증가 → Minor GC 비용이 비선형으로 증가하다 임계점을 넘으면, Flush 콜백이 GC 뒤에 밀리면서 피드백 루프가 형성됩니다.

  프로파일링 필요 여부
                                                                                                                                                                                
  CPU 프로파일링: 불필요 — CPU가 병목이 아닙니다.
                                                                                                                                                                                
  Heap 프로파일링: 권장 — 두 가지를 확인해야 합니다:
                                                                                                                                                                                
  1. 무엇이 Old gen에 누적되는가 — 14MB 증가의 원인 (게임 내내 살아남는 객체: Room 상태, Player 세션, Redis 응답 캐시 등)                                                       
  2. Position Flush 경로의 임시 객체 할당량 — 200명 위치를 직렬화할 때 생성되는 string/array 규모
                                                                                                                                                                                
  Pyroscope가 datasource에 있으니 해당 부하테스트 시간대를 바로 조회할 수 있습니다.
