## 발견 1: RxJS 구독 체인 누적 — Old gen 비대화의 주범

### 증거

Context 객체에 캡처된 변수 상위 목록:

```
subscriber   : 5,565
onError      : 3,746
onComplete   : 3,642   ← OperatorSubscriber 수와 정확히 일치
onNext       : 3,642
destination  : 3,642
buffer       :   913
concurrent   :   912   ← 아래 6개 전부 912로 일치
expand       :   912
isComplete   :   912
checkComplete:   912
outerNext    :   911
doInnerSub   :   911
```

`buffer`, `concurrent`, `expand`, `isComplete`, `checkComplete`, `outerNext`, `doInnerSub`는 RxJS `mergeMap` 연산자 내부 변수명이다.
912개가 동시에 살아있다는 것은 **mergeMap 파이프라인 912개가 완료되지 않은 채 Old gen에 잔류**함을 의미한다.

### 객체 보유 구조

```
Array
 └─ _finalizers → SafeSubscriber
                    └─ _parentage → OperatorSubscriber  (×3,642)
                                      ├─ _next   → [closure]
                                      ├─ _error  → [closure]
                                      └─ _complete → [closure]
                                                       └─ Context (subscriber / destination 캡처)
```

### 원인

NestJS의 `@SubscribeMessage()` 핸들러가 Observable을 반환할 때 프레임워크가 내부적으로 `mergeMap`으로 감싸 구독한다.
200명 플레이어 × ~4.56개 구독 ≈ 912개.

**플레이어가 접속 중인 동안에는 이 구독들이 살아있으므로 누수 자체는 아니다.**
그러나 구독 체인 전체(closure + Context + OperatorSubscriber)가 Old gen에 올라가
**Minor GC의 루트 셋을 비대화**시켜 루트 스캔 비용을 선형 이상으로 증가시킨다.

## 발견 2: Position Flush 배열 할당 — Young gen 주범

`[array]` +7,998개, +1.53 MB 증가 확인.

`position-broadcast.service.ts`의 `execFlushLua` 내부에서 flush마다 5개 배열을 새로 생성:

```typescript
const playerKeys: string[] = new Array(n);
const posXArgs: string[]   = new Array(n);
const posYArgs: string[]   = new Array(n);
const isAliveArgs: string[] = new Array(n);
const playerIdArgs: string[] = new Array(n);
```

200명 × 10회/s(100ms 주기) = 초당 배열 1,000개 생성. 대부분 Young gen에서 회수되지만
일부가 오래 살아남아 Old gen으로 승격되면서 누적된다.

추가 할당 경로:
- `mergeBatches`의 spread 연산자 `[...aliveMap.values()]` — flush마다 중간 배열 2개 생성
- alive 플레이어당 `{ playerId, playerPosition: [x, y] }` 객체 리터럴 생성

## 발견 3: Socket 객체 +319개 — 정상 범위

Socket.IO `Client → Map(sockets)` 경로로 정상 보유. 200명 플레이어 + Socket.IO 내부 오버헤드(admin, game 네임스페이스 등) 수준.

Socket 일부가 RxJS Observable의 `_subscribe` 클로저에 `target`으로 캡처되어 있으나,
이는 NestJS가 WebSocket 이벤트를 Observable로 감쌀 때 생기는 정상 패턴이다.

---

## 발견 4: 대형 문자열 — 문제 없음

t7 상위 대형 문자열 전부가 `lodash`, `BSON`, TypeORM 소스코드 등 **모듈 로드 시 생성된 정적 문자열**이다.
게임 진행 중 동적으로 생성된 것이 아니므로 GC 압력과 무관하다.

---

## GC 압력 인과 관계 (업데이트)

```
200명 접속
 → NestJS WebSocket 프레임워크가 플레이어당 mergeMap 구독 체인 생성
 → 200명 × ~4.56 = ~912개 closure+Context+OperatorSubscriber → Old gen 점유 (~2.56 MB)
 → Old gen 비대화 → Minor GC 루트 스캔 비용 증가

200명 위치 업데이트 (100ms 주기)
 → flush당 5개 배열 × 200명 → 초당 배열 1,000개 생성 → Young gen 빠르게 채움
 → Minor GC 빈도 증가 (6.5/s)

두 원인 결합 → 후반부 비선형 악화
 → Old gen 비대화로 루트 스캔 느려짐 + Young gen 압력 지속
 → Minor GC 1회 pause 0.6ms → 9ms+
 → Event Loop Lag P99: 11ms → 500ms
 → Position Flush P99: 5ms → 474ms
```

## 개선 우선순위

| 우선순위 | 대상 | 방법 |
|---|---|---|
| 1 | flush당 5-array 생성 | `execFlushLua` 배열 재사용 또는 flat 배열 하나로 통합 |
| 2 | `mergeBatches` spread 제거 | Map iteration 결과를 직접 push |
| 3 | RxJS 구독 수 모니터링 | 게임 종료 시 active 구독 수 Prometheus 지표화 |
| 4 | debug 로그 guard | `logger.isDebugEnabled()` 조건 추가로 불필요한 `.map()` 제거 |
