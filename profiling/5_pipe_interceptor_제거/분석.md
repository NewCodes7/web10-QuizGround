## 게임 종료 스파이크 (T+5.5 ~ 6.5분) — 새로 드러난 병목

이전 테스트에서는 프로파일링 오버헤드에 가려져 있던 병목이 이번에 명확하게 드러났다.

### Event Loop Lag P99

```
node1:  116ms → 153ms → 149ms → 162ms   (40초 지속 → 즉시 11ms 복귀)
node2:  238ms → 256ms → 162ms            (node2가 훨씬 심각)
```

### Node CPU

```
node1:  19% → 26% → 36% → 44% → 43%    (게임 호스팅 담당 인스턴스)
node2:  11% → 22% → 25%
```

### Position Flush P99 (히스토그램 artifact)

```
81ms → 242ms → 310ms → 380ms → 292ms
```

실제 지연이 아니라 스파이크 구간 데이터가 1분 슬라이딩 윈도우에 남아있는 잔상이다.  
이벤트루프가 11ms로 복귀한 후에도 P99가 계속 올라가는 것이 그 증거.

### Redis 명령 패턴 변화

| 커맨드 | 게임 중 | 게임 종료 시 |
|--------|---------|------------|
| `hmset` | ~340 ops/s | ~272 ops/s (포지션 업데이트 감소) |
| `hgetall` | ~0 | **8.9 ops/s** 급증 |
| `hget` | ~0 | **4.4 ops/s** 급증 |

→ 게임 종료 시 **DB archive + Redis cleanup** (hgetall로 room 전체 읽기 → srem/del로 정리)이  
이벤트루프에서 동기적으로 실행되면서 40초간 블로킹하는 것으로 추정된다.

## Nginx CPU 100% → 70% 원인 분석

**프로파일링 ON (이전 테스트)**  
Node.js CPU 샘플링이 V8 내부에서 추가 syscall/signal 발생 → Nginx(같은 VM)가 그 오버헤드까지 처리 → softirq 폭발 → 100%

**프로파일링 OFF (이번 테스트)**  
Node.js 오버헤드 제거 → Nginx softirq 감소 → 70%

Prometheus Nginx softirq는 코어별 ~7~10% 피크로, Grafana 패널에서 모든 non-idle 모드를 합산하면 70% 수준으로 표시된다.

## 남은 병목 (우선순위 순)

### 1순위: 게임 종료 처리 이벤트루프 블로킹 (EL 256ms)

추정 흐름:
```
gameEnd
  → hgetall(Room:{gameId})    # room 전체 읽기
  → DB archive (TypeORM)      # I/O지만 await 체인이 길면 EL 점유
  → Redis cleanup             # srem / del 다수
```

개선 방향:
- cleanup 작업을 `setImmediate` 체인으로 분산하여 이벤트루프를 주기적으로 양보
- DB archive를 별도 async queue(BullMQ 등)로 이전하거나, 최소한 `Promise.all` 병렬 처리

### 2순위: GC — node2가 node1보다 심함

- node2 major GC avg: **6~14ms** (node1 대비 높음)  
- node2 minor GC: 게임 종료 구간에서 3.7ms까지 상승  
- GC 비대칭이 EL lag 차이(node2 256ms vs node1 162ms)에 직접 기여

### 3순위: `util.deprecate` 경고 (~4.5% CPU)

**원인 확인됨 (4번 프로파일 콜스택 분석)**

정확한 호출 경로:
```
socket event (updatePosition / chatMessage)
  → interceptors-consumer.js:20 handle()
    → AsyncResource.bind(fn)          [async_hooks:266]
      → AsyncResource.prototype.bind  [async_hooks:230]
        → util.deprecate() × 2        [node:internal/util:146]  ← 4.5% CPU
```

`AsyncResource.prototype.bind` 내부에서 `bound` 함수를 반환할 때마다 `util.deprecate()`를 **매번 2번 호출**해서 `asyncResource` 프로퍼티 getter/setter wrapper를 새로 생성한다 (DEP0172).

```js
// Node.js async_hooks.js — AsyncResource.prototype.bind 내부
ObjectDefineProperties(bound, {
  'asyncResource': {
    get: deprecate(fn, 'The asyncResource property on bound functions is deprecated', 'DEP0172'),
    set: deprecate(fn, 'The asyncResource property on bound functions is deprecated', 'DEP0172'),
  },
});
```

경고 출력은 1회지만 `deprecate()` 클로저 생성 비용이 **매 호출마다** 발생한다.

| 항목 | 수치 |
|---|---|
| `AsyncResource.bind` 1회 비용 | ~5,500ns (~6μs) |
| 직접 함수 래핑 비용 | ~50ns |
| 오버헤드 배율 | ~110배 |
| 4번 테스트 hot path 빈도 | ~8,000~12,000 calls/s (updatePosition × 200명 × interceptor 2개) |

**현재 상태**: 5번 테스트(핫패스 interceptor 제거)로 이 코드 경로 자체가 없어졌다. 5번 프로파일에서 `deprecate` self-time이 ~30ms(0.0%)로 사실상 소멸 — 이미 해소됨.

### 4순위: Nginx softirq 부하

- 2코어 합산 softirq ~14~20% (게임 중)
- TCP 인터럽트 처리 비용이 주된 원인

#### softirq 원인 구조

Nginx가 WebSocket 리버스 프록시로 동작할 때 TCP 연결이 이중으로 생성된다.

```
클라이언트 ──TCP 1──▶ nginx ──TCP 2──▶ node-1/node-2
              ↑ 인터럽트       ↑ 인터럽트
         softirq 발생 2회/패킷
```

Socket.IO 위치 업데이트가 초당 ~2,000 패킷 발생 → nginx VM에서 초당 ~4,000 TCP 인터럽트 → e2-micro 1vCPU 기준 14~20% softirq.

#### Nginx 역할 목록 (제거 시 대체 필요)

| 역할 | 현재 Nginx | 대체 가능 여부 |
|---|---|---|
| FE 정적 파일 서빙 | `root /var/www/dist` | GCS + Cloud CDN으로 대체 가능 |
| SSL 종료 (HTTPS/WSS) | Let's Encrypt + certbot | 직접 Node.js에 cert 적용 또는 GCP Load Balancer |
| 로드밸런싱 | upstream round-robin | GCP Cloud Load Balancer 또는 Node 직접 노출 |
| Sticky session | `cookie SERVERID` | GCP LB cookie affinity 또는 세션 불필요 설계 |
| WAS external IP 차단 | 내부망 전용, nginx만 외부 IP | WAS에 직접 external IP 부여 시 방화벽 필요 |
| Rolling deploy | nginx를 통해 node-1/2 순차 재시작 | Blue-green 또는 GCP Instance Group |
| CI/CD 배스쳔 | nginx VM을 통해 내부망 접근 | Cloud IAP 또는 Bastion VM 별도 운영 |

#### 대안별 트레이드오프

**A. 현 상태 유지 + Nginx 설정 튜닝**

```nginx
worker_processes auto;          # vCPU 수만큼 워커 생성
worker_cpu_affinity auto;       # 코어 고정으로 캐시 효율 개선
```

- `worker_cpu_affinity auto`는 e2-micro 1vCPU에서 효과 없음(코어가 1개)
- e2-micro → e2-small(2vCPU) 업그레이드 시 ~50% softirq 분산 가능
- 비용 증가: ~$7/월

**B. FE를 GCS+CDN으로 분리, Nginx는 WS 프록시 전용**

```
클라이언트
  ├── FE 정적 파일 → GCS (Cloud CDN)    ← nginx FE 트래픽 완전 제거
  └── /game WebSocket → nginx → WAS
```

- nginx 패킷 수 ~30~40% 감소 (정적 파일 요청 제거)
- softirq 10~14%대로 감소 예상
- 비용: GCS 거의 무료 수준 (정적 파일 소량)
- 난이도: **낮음** — 배포 스크립트 수정만 필요

**C. GCP HTTP(S) Load Balancer로 교체**

```
Internet → GCP LB (관리형) → node-1 / node-2
```

- softirq 문제 근본 해소 (nginx VM 자체 제거)
- SSL, sticky session, 헬스체크 모두 GCP가 관리
- **단점**: WebSocket(Upgrade 헤더) 지원 확인 필요, 월 ~$18 고정비, 디버깅 복잡
- Rolling deploy 방식 변경 필요 (Instance Group rolling update)
- Sticky session: GCP LB는 HTTP cookie affinity 지원하나 WebSocket 재연결 시 동작 검증 필요

**D. Nginx 완전 제거, node-1/node-2 직접 노출**

```
Internet → node-1 (external IP, :3000)
         → node-2 (external IP, :3000)
```

- softirq 완전 제거
- **단점 (치명적)**:
  - WAS가 인터넷에 직접 노출 → 방화벽 규칙 직접 관리
  - SSL: Node.js에 cert 적용 필요 (자동 갱신 별도 구성)
  - 클라이언트가 어느 노드로 접속할지 알아야 함 (로드밸런싱 없음)
  - Rolling deploy 시 DNS 전환 또는 IP 변경 필요
  - 현재 sticky session 구조 붕괴 → Redis 기반 세션은 유지되나 Socket.IO room 재구성 필요

#### 권장 방향

**단기 (지금 바로)**: B안 — FE를 GCS+CDN으로 분리  
구현 비용이 낮고 softirq를 즉시 줄이면서 Nginx 역할은 유지.

**중기 (필요 시)**: e2-micro → e2-small 업그레이드 또는 C안(GCP LB) 검토  
현재 softirq 14~20%는 게임 플레이를 직접 방해하는 수준은 아님. Node CPU 12~13%, EL lag 11~14ms가 안정권이므로 Nginx softirq가 새로운 병목이 될 때 재검토.

**D안은 비추**: 보안/운영 복잡도 증가가 성능 이득보다 크다.
