# msgpackr 적용 — CPU 프로파일 3개 비교 분석

> 작성일: 2026-05-02  
> 프로파일: 기존(JSON) / msgpack-notepack.io / msgpackr

## 프로파일 3개 비교 요약

| 항목 | 기존(JSON) | msgpack (notepack.io) | msgpackr |
|------|-----------|----------------------|----------|
| 파일 | `기존_cpu_heap_profiling/cpu-2026-04-30T13-01-56.cpuprofile` | `msgpack_적용_cpu_profiling_260502/cpu-2026-05-02T03-04-05.cpuprofile` | `msgpackr_적용/cpu-2026-05-02T07-51-47.cpuprofile` |

| 총 샘플 수 | 3,118 | 1,520 | 4,728 |
| **Idle** | 28.9% | 8.7% | 27.7% |
| **GC (전체 기준)** | 8.7% | 18.8% | 8.6% |

> GC active 기준 = `GC 샘플 / (전체 샘플 - Idle 샘플)`  
> 실제 CPU가 일하는 시간 중 GC가 차지하는 비율이므로 이 수치가 더 의미 있음.

## 해석

### msgpack (notepack.io)가 오히려 나빴던 이유

notepack.io는 `encode()` 호출마다 `Buffer.allocUnsafe(size)` 를 새로 할당하고 버린다.
200명 방 × 초당 10회 브로드캐스트 = 초당 2,000개의 단수명(short-lived) Buffer 생성 → GC 대상 폭증.

결과: GC active 기준 **12.2% → 20.5%** 로 악화. Idle도 28.9% → 8.7%로 줄어들어 서버 전체가 훨씬 바빠진 상태.

### msgpackr 결과

- GC: **20.5% → 11.9%** 로 기존 JSON 수준 복원 → notepack.io GC 문제 해결 성공
- Idle: 8.7% → 27.7%로 회복 → 서버 전체 부하도 정상화
- 단, msgpackr encode 자체가 active CPU의 **22%** 소비
  - 기존 JSON의 `encodeAsString + hasBinary` (~19%) 제거분을 msgpackr encode가 대체하는 구조
  - `hasBinary` 재귀 순회 제거, UTF-8 변환 제거라는 이점이 있으나 encode 비용으로 상쇄됨
  - 체감 개선이 적은 이유

### msgpackr 적용 후 남은 병목 (active CPU 기준)

| 항목 | active CPU | 비고 |
|------|-----------|------|
| msgpackr encode | ~22% | `pack` + `writePlainObject` |
| GC | ~12% | 기존 수준 복원 |
| writev (net I/O) | ~10% | 소켓 전송 자체 |
| admin-ui | ~4% | 부하테스트 시 오염 요소 → 이번에 제거 완료 |
| class-validator | ~2% | position update hot path에 여전히 살아있음 |

## 내 분석에 대한 답변

### Q. msgpackr도 성능 개선이 시원찮음

정확한 관찰이다. 단, 이유는 두 가지로 나뉜다.

1. **측정 자체가 오염된 테스트였음**: 해당 테스트 중 node-2가 크래시(07:50:44 TCP RST)했고, 이 때문에 플레이어들이 튕기고 nginx가 과부하 상태였다. 07:51:47 프로파일은 혼란이 가라앉은 직후 수집된 것.
2. **encode 비용이 hasBinary+UTF8 비용을 대체**: 제거한 비용만큼 새 비용이 생긴 구조라 순수 비교 시 극적인 차이가 없음. 진짜 이점은 payload 크기 감소(네트워크 절감)와 GC 정상화.

### Q. nginx가 100%를 찍은 원인

nginx 로그(`recv() failed (104: Connection reset by peer)`) 분석 결과:

```
07:50:44  node-2 프로세스 크래시 → OS TCP RST 전송
          → node-2에 연결된 ~100명 WebSocket 동시 끊김
          → socket.io auto-reconnect 즉시 발동
          → ~100명이 동시에 nginx에 WebSocket 업그레이드 요청 폭주
          → nginx CPU 100%
07:51:25  node-1도 과부하(2배 연결 갑자기 떠안음) → 양쪽 다 temporarily disabled
          → 브라우저 polling이 502 받고 즉시 재시도 → no live upstreams 연속 10회
```

**binary vs JSON이 nginx CPU에 영향을 준 게 아님.** nginx는 WebSocket 업그레이드 이후 TCP relay만 하므로 payload 포맷(text/binary)은 nginx CPU와 무관하다.

원인은 서버 크래시로 인한 재연결 폭풍이고, nginx는 피해자다.

### Q. nginx를 통하지 않고 직접 연결해야 하나

트레이드오프 정리:

| 항목 | nginx 유지 | nginx 우회 (직접 노출) |
|------|-----------|----------------------|
| 보안 | 내부 IP 보호, TLS 한 곳에서 관리 | 노드 VM이 직접 인터넷 노출 |
| 로드밸런싱 | nginx에서 처리 | 클라이언트가 직접 노드 선택하거나 DNS 기반 |
| sticky session | cookie 기반 쉽게 구현 | 직접 구현 필요 |
| 배포 | 롤링 배포 시 nginx에서 upstream 전환 | 배포 전략 직접 구현 |
| 장애 격리 | nginx가 down된 노드 자동 제외 | 클라이언트 재시도 로직 직접 구현 |
| CPU 오버헤드 | WebSocket 프록시 CPU 추가 | 제거 |

현재 아키텍처(분산 WAS 2대, 롤링 배포, sticky session)에서 nginx를 제거하면 얻는 게 CPU 약간인데 잃는 게 너무 많다. 현실적인 선택이 아님.

### Q. 230명으로 더 만들어진 이유

node-2 크래시 → auto-reconnect → 재연결 시도 중인 클라이언트가 새 세션으로 중복 집계된 것으로 추정. 실제 unique 클라이언트는 200명이었을 것.

### Q. GC 근본 옵션

Node.js 플래그로 GC를 직접 튜닝할 수 있으나 대부분 임시방편이다.

```bash
# young gen 크기 증가 → minor GC 빈도 감소 (기본 16MB)
node --max-semi-space-size=64 server.js

# old gen 증가 → major GC 빈도 감소
node --max-old-space-size=1536 server.js
```

더 효과적인 방법은 할당 자체를 줄이는 것:
- `packr.useBuffer(Buffer.allocUnsafe(2MB))` → encode 작업 버퍼 arena 재사용 (이번에 적용 완료)
- hot path ValidationPipe 제거 → position update마다 생성되는 DTO 객체 감소
- 브로드캐스트 빈도 조절 → encode 호출 횟수 자체를 줄임

GC 플래그보다 **할당 횟수를 줄이는 코드 변경**이 항상 더 근본적이다.
